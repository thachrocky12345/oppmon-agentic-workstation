# TAG-64: Integration Tests + Eval Entries

## Description

**Suggested Points:** 3
**Type:** Story / QA
**Epic:** [TAG-50](./TAG-50-authenticated-solve-endpoint-epic.md)
**Status:** Open

End-to-end coverage that proves the epic landed without regressing `/solve_v2`
and with the four critical security/correctness gates passing.

## Objective

A `pytest` suite + a `evals/` extension that, together, gate the epic merge.

## Requirements

### Integration suite (real DB, fake LLM)

`tests/integration/test_solve_e2e.py` — uses docker-compose `postgres` + a
seeded fixture corpus, plus the `FakeLLMClient` from TAG-49's factory.

Test cases:

| # | Scenario | Pass condition |
|---|---|---|
| 1 | `POST /solve_v2` unchanged | Same SSE frames as a pre-epic snapshot |
| 2 | `POST /solve` no auth | 401 |
| 3 | `POST /solve` bad JWT | 401 |
| 4 | `POST /solve` model not owned (cross-tenant) | 403, no model existence side-channel |
| 5 | `POST /solve` `webFallback=false collectionIds=[]` | 422 |
| 6 | `POST /solve` `webFallback=false collectionIds=[owned]` | 200 SSE, citations `[[doc:chunk]]` |
| 7 | `POST /solve` `webFallback=false collectionIds=[owned]` w/ OUT-OF-CORPUS question | refusal string verbatim |
| 8 | `POST /solve` `webFallback=true collectionIds=[]` | matches `/solve_v2` for identical question |
| 9 | `POST /solve` hybrid: corpus answers fully | NO web mock called |
| 10 | `POST /solve` hybrid: corpus partial | web mock called for unanswered sub-Q only |
| 11 | Plaintext API key absent from any captured log/trace | `grep -E "sk-|csk-"` of captured logs returns nothing |
| 12 | Decrypted secret never appears in SSE `error` event | inject a forced error mid-stream, inspect frame |

### Eval extension

Add to `evals/questions.json` (or create `evals/corpus-questions.json` if the
existing harness only handles web):

- 5 corpus-grounded questions with known answers from a seeded corpus.
- 5 out-of-corpus questions where the expected behavior is refusal.

LLM-judge gates:

- Citation presence ≥ 0.95 on grounded questions.
- Refusal rate ≥ 0.90 on out-of-corpus questions.
- No regression vs `/solve_v2` baseline on existing web evals.

### Captured fixtures

`tests/integration/fixtures/`:

- `seed_two_tenants.sql` — two tenants, two users, two models per user, one
  collection per user with a small corpus.
- `seed_models_with_ts_encryption.json` — model rows whose `secret_vault` was
  encrypted by `apps/api/src/crypto/secret-vault.ts` (use the helper from
  TAG-54) so the round-trip is real.

### CI wiring

`.github/workflows/agent-search-tests.yml` (or extend existing):

```yaml
- name: Seed fixtures
  run: psql $DATABASE_URL -f tests/integration/fixtures/seed_two_tenants.sql
- name: Run integration tests
  run: pytest tests/integration -q
- name: Run evals
  run: node evals/scripts/run.ts --suite corpus-rag
- name: Grep-test for leaked secrets
  run: ./scripts/check-no-secrets-in-logs.sh ./test-artifacts/
```

## Implementation Notes

- Use FakeLLM with a scripted response set so the integration suite is
  deterministic. Real-LLM eval runs are gated to the evals job (slower, costs
  money).
- The grep-test is the last line of defense against the `decrypt → log` mistake.
  Run it on `pytest --capture=tee-sys` output too.
- Snapshot of `/solve_v2`'s SSE frames goes in `tests/integration/snapshots/`.
  Re-record only with `UPDATE_SNAPSHOTS=1`.

## Acceptance Criteria

- [ ] All 12 integration tests pass.
- [ ] Citation gate ≥ 0.95 passes.
- [ ] Refusal gate ≥ 0.90 passes.
- [ ] `/solve_v2` snapshot unchanged.
- [ ] Secret-grep test green.
- [ ] CI workflow gated as required on the protected branch.

## Risk Factors

| Risk | Mitigation |
|---|---|
| Flaky LLM-judge eval | Use a deterministic judge prompt; record judge scores per question for stability tracking. |
| Fixture drift vs Prisma schema | Seed SQL is generated by a script that introspects `schema.prisma`; CI regenerates. |
| Snapshot churn from minor planner output drift | Snapshot covers the SSE event TYPES and ORDER, not full content. |
